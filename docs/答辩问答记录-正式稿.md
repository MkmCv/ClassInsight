## 一、视频上传的限制是什么？

**答：**

本系统对课堂录像上传采用「格式白名单 + 单文件体积上限」的双重约束，并在前端与后端配置中保持同一数量级说明。

**（1）单文件体积上限**  
- **前端（Streamlit）**：在 `config.toml` 中设置 `maxUploadSize = 2048`，单位为 **MB**，即单文件约 **2GB**。  
- **后端配置**：`Settings.MAX_UPLOAD_SIZE` 定义为 **2 × 1024³ 字节（2GB）**，可通过环境变量或配置文件覆盖，便于在不同部署环境中调节。  
- **接口与文档**：OpenAPI 及前端上传页文案均标明最大约 **2GB**，与上述配置一致。

**（2）允许的视频格式**  
当前允许扩展名为 **`.mp4`、`.avi`、`.mov`、`.mkv`**，与 `ALLOWED_VIDEO_EXTENSIONS` 一致；不在列表内的格式将在上传阶段被拒绝并提示用户。

**（3）工程与部署层面的补充说明**  
超大文件除受「2GB」配置约束外，还可能受 **HTTP 反向代理 body限制、请求超时、服务器内存与磁盘** 等因素影响；生产环境建议在网关层与接口层增加**流式写入与显式体积校验**，避免整文件读入内存带来的风险。答辩中若被问及「后端是否逐字节强校验」，可如实说明：设计上已定义 `MAX_UPLOAD_SIZE`，具体校验策略可按部署阶段加固。

## 二、目前上传的视频大小一般是多少？是否可以调节？

**答：**

**系统并未在代码中写死「典型上传体积」**。实际上传大小完全取决于**录制设备码率、分辨率、帧率与课时时长**：例如一节 40～45 分钟的1080p 课堂录像，常见在数百 MB 至 1GB 以上均属正常；答辩演示中使用约 **900MB** 的样例仅反映某次实测或演示素材，**不具有全局代表性**。

**调节方式**包括：  
- 调高或调低 **Streamlit `maxUploadSize`** 与 **后端 `MAX_UPLOAD_SIZE`**，使产品层上限与运维能力匹配；  
- 在采集端通过**降低码率、分辨率或分段录制**控制单文件体积；  
- 在部署层调整 **代理与超时**，避免大文件传输中断。

## 三、系统如何识别「教学模式」？

**答：**

教学模式**不是**由单帧图像直接输出的单一类别，而是**在整段视频分析流水线末端**，由**结构化行为统计 + 确定性规则**共同给出的**时间窗级标签**。

**（1）上游：检测与聚合**  
系统使用 **多个 YOLO-vHeat 检测模型**（综合场景、学习行为、讨论、姿态等）对采样帧进行目标检测，将结果映射到统一的全局类别体系（如 `guide`、`answer`、`blackboard-writing`、`teacher`、`screen`、学生 `read` / `write` / `discuss` 等）。随后按固定**非重叠时间窗**对各类检测计数进行聚合，得到每个时间窗内的 **`behavior_counts`**（各类行为出现强度或频次的统计表示）。

**（2）下游：教学模式规则**  
在 `behavior_analyzer` 中，函数 **`identify_teaching_mode(behavior_counts)`** 根据当前窗口内多类行为的相对强弱与优先级，输出离散教学模式，包括但不限于：**互动教学、板书讲解、多媒体演示、练习模式、讲授模式、其他** 等。  
该过程**综合**了教师侧行为（如讲授、指导、应答、台上互动）、场景元素（板书、大屏）、以及学生侧行为（阅读、书写、讨论）的统计信息，而非简单「一种教师行为对应一种模式」的一一映射。

**（3）与「整课报告」的关系**  
对全课所有时间窗的模式序列可进一步统计**模式占比、模式转换次数与转换路径**，用于课堂结构分析与可视化展示，支撑教学反思与督导场景。

## 四、教学模式的判断是否由大模型完成？

**答：**

**不是。** **教学模式标签**由后端分析服务中的 **Python 规则逻辑**（如 `identify_teaching_mode` 及相关的教学模式聚合函数）**确定性地计算**得出，并通过行为分析相关 API 返回给前端；其输入为已落库的**检测结果与时间线统计**，输出为**可复现、可审计**的符号标签与占比。

**大模型**在本系统中的定位是 **教学建议与多轮对话顾问**：在 **LangGraph** 工作流中，通过 **通义千问（DashScope）** 读取已为该视频准备好的**摘要、时间线、异常事件**等上下文，生成自然语言层面的**解读、追问与改进建议**。因此应明确区分两条链路：  
- **分析链路（判别式）**：YOLO + 时间聚合 + 规则 → 教学模式与统计数据；  
- **顾问链路（生成式）**：结构化分析结果 + 大模型 → 可读报告与对话。

二者串联使用，**职责分离**，避免将「模式分类」与「文案生成」混为一谈，也便于答辩中回应「可解释性」「是否黑盒」等问题。

## 五、规则引擎是附着在 YOLO 上，还是独立实现？与检测模型是什么关系？

**答：**

**规则引擎不附着在 YOLO 权重或推理图内部**，也**不参与反向传播训练**。YOLO 的职责仅限于：**从图像中预测边界框与类别（及置信度）**；规则引擎位于 **检测完成之后的业务层**，对**已结构化的计数与时间序列**进行逻辑判断。

**（1）数据流关系**  
可概括为：**视频帧 → 多模型 YOLO 推理 → 单帧或多帧检测结果 → 时间窗聚合（`behavior_counts`）→ 教学模式规则 / 行为分类规则 → 高层语义与报表**。  
因此，规则引擎的输入是 **YOLO 分析后的结构化数据**（及中间分类结果），而非原始像素。

**（2）与示例表述的对应**  
答辩中举例「板书讲解」时，更贴近代码逻辑的说法是：当某时间窗内 **`blackboard-writing` 等板书相关统计满足规则条件**时，可判定为 **板书讲解**；「互动教学」则与 **`guide`、`answer`、`On-stage interaction` 等互动相关计数**以及**讨论类行为**的综合判断有关，而不是机械拼接字符串「teacher + blackboard_write」。  
此外，**师生复合行为**还可通过 **`behavior_classifier`** 中基于 required / exclude 等字段的规则表进行细化，与教学模式规则同属**后处理层**，便于独立迭代而无需重新训练检测网络。

**（3）架构优势**  
将规则与模型解耦，有利于：**快速调整教学评价口径**、**对接不同课程标准**、**在同等检测输出上试验不同聚合策略**，同时保持深度学习模块相对稳定。

## 六、改进 YOLO12（结合 vHeat）的主要目的是什么？

**答：**

课堂视频具有**视角固定但场景复杂、人物尺度变化大、遮挡与相似动作多**等特点，对检测器的**鲁棒性与定位精度**要求较高。本工作在 **YOLO12类检测框架**基础上引入 **vHeat 相关结构（如热传导式算子等）**，旨在增强特征提取中对**长程依赖与局部细节**的建模能力，从而在**噪声背景、光照变化、部分遮挡**等条件下仍保持较稳定的框回归与分类表现。

改进目标可归纳为：  
- **鲁棒性**：减少因环境变化导致的漏检与类别抖动；  
- **精度**：在相同或相近数据规模下，提升关键教育行为类别的检测质量；  
- **与工程一致**：训练在独立 **yolo-vheat** 工程中完成，部署时以 **Ultralytics 兼容管线**加载多权重，与本系统 **视频处理与时间聚合**模块对接。

具体网络改动与实验数据以学位论文及 `Model components/yolo-vheat` 相关说明为准。

## 七、目前模型识别精度如何？是否开展消融实验？COCO128 对比如何表述？

**答：**

**（1）精度与局限（课堂场景）**  
在 **SCB 系列等教育场景数据**上训练得到的专用模型，对**常见、尺度适中、标注语义清晰**的行为类别整体可用，能够支撑**整课统计、时间线、教学模式粗粒度划分**等业务。  
受 **数据集规模、类别不平衡、细粒度动作难以区分** 等因素限制，在**学生小动作、远距离小目标、严重遮挡、强相似行为**等情形下，**误检与漏检仍较明显**，这是当前课堂行为检测领域的共性瓶颈，并非单靠替换骨干即可完全消除。

**（2）系统侧可调因素**  
除模型权重外，**置信度阈值、NMS IoU、视频抽帧密度**等配置会同时影响**召回率、精度与处理耗时**；答辩中可说明系统在这些参数上保留了**工程可调空间**，以便在「更稳的统计」与「更细的时序」之间权衡。

**（3）消融实验**  
**未开展严格的模块级消融实验**（例如系统化地去掉 vHeat 块、逐层替换并在同一课堂验证集上报告 mAP 变化）。若评委追问，应**诚实说明**，并可将后续工作表述为：在统一课堂验证集上补充 **vHeat / 非 vHeat、不同训练日程** 的对照实验。

**（4）COCO128 与原版 YOLO12 对比**  
若在 **COCO128** 上与 **原版 YOLO12** 进行过实验，宜表述为：**轻量级、快速可复现的一致性检查或迁移验证**，用于观察改进结构在通用小集上的训练/推理行为是否正常；**不能**将其等同于本系统在**全量课堂数据**上的性能结论，更**不能**替代针对 **vHeat 模块**的正式消融。答辩中主动界定实验范围，反而体现严谨性。
