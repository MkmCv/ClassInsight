# ClassInsight AI 课堂行为智能分析系统 — 毕业答辩陈述稿

> 使用说明：全文可按 **8～12 分钟** 陈述裁剪；**§1～§9** 为主干，**§10** 为演示顺序，**§12** 备答辩提问。口头报告时可将加粗处作为「必讲句」。

## §1 开场（约 30 秒）

各位老师好。我是×××，毕业设计题目是 **《ClassInsight AI——课堂行为智能分析系统》**。

本设计面向中小学或师范场景中的 **课堂录像**，用计算机视觉与数据分析辅助教师 **回顾课堂、理解师生行为分布，并获得可操作的改进提示**。下面从背景、方案、实现与演示四个方面汇报。

---

## §2 研究背景与意义（约 1 分钟）

**背景**：课堂观察是教研与教学改进的重要环节，但完全依赖人工回看录像 **耗时、主观性强**，难以对整节课的 **互动、专注、参与** 等做稳定、可对比的量化描述。

**意义**：本系统将 **视频 → 行为检测 → 时间聚合 → 统计与可视化 → 规则化建议与大模型顾问** 串成闭环，使教师能在 Web 界面中 **快速把握课堂节奏**，并为 **教学反思与答辩展示** 提供数据支撑。系统采用 **Python 全栈**，便于与现有教研环境中的算法与数据格式对接。

---

## §3 需求与目标（约 1 分钟）

**功能目标**可概括为：

1. **用户与权限**：教师、管理员、超级管理员分级；登录、注册、安全策略与审计。
2. **视频与任务**：上传课堂录像，后台异步分析，展示处理进度与元数据。
3. **行为分析**：多类别检测（师生动作、场景元素等），**时间轴上的统计**、异常提示、教学模式相关展示。
4. **教学优化**：基于分析结果的 **结构化改进建议**，以及可选的 **AI 多轮对话顾问**。
5. **课表与运维**：课表管理、首页日历；超管侧安全与模型相关配置。

**非功能目标**：前后端分离、接口可文档化（OpenAPI）、单机可部署（默认 SQLite）、核心流程可演示、关键逻辑可解释（规则 + 数据来源明确）。

---

## §4 总体架构（约 1.5 分钟）

**一句话技术方案**：**Streamlit 多页面前端 + FastAPI 异步后端 + SQLite，REST API 与 JWT 认证；课堂侧用 YOLO-vHeat 多模型检测，教学建议侧用规则引擎与通义千问（LangGraph 编排）相结合。**

**逻辑分层**：

- **表现层**：两个 Streamlit 应用——**普通系统**（教师/管理员，默认端口 **8501**）与 **超级管理员系统**（**8502**）。
- **服务层**：**FastAPI**（默认 **8000**），异步路由、依赖注入、统一异常与鉴权。
- **领域层**：视频处理、多模型检测封装、行为分类、时间窗聚合、分析指标与教学优化规则、Agent 工具读库。
- **数据层**：**SQLAlchemy 异步 ORM**，默认 **SQLite** 单文件（`backend/storage/classinsight.db`），启动时 `create_all` 建表。

**数据流（主干）**：用户登录获得 Token → 上传视频 → 后端入队/异步处理 → 抽帧调用检测器 → **按固定秒级时间窗聚合** → 写入时间线与汇总表 → 前端图表与建议接口、Agent 拉取同源数据生成对话。

---

## §5 技术选型说明（约 1 分钟）

| 层次 | 选型 | 理由（答辩口径） |
|------|------|------------------|
| 前端 | Streamlit | Python 与数据科学栈一致，快速构建多页面管理端与 Plotly 图表，适合毕设原型与演示。 |
| 后端 | FastAPI + Uvicorn | 异步 I/O、自动生成 API 文档、类型提示与 Pydantic 校验，利于前后端协作。 |
| 数据库 | SQLite（可换） | 零独立数据库服务、部署简单；ORM 抽象便于迁移到 PostgreSQL 等。 |
| 视觉 | YOLO-vHeat / Ultralytics | 多权重协同检测课堂相关类别，与项目 `Model components/yolo-vheat` 集成。 |
| 智能顾问 | LangGraph + 通义千问 | 将「汇总、时间线、异常」注入 Prompt，支持多轮教学场景问答。 |
| 安全 | JWT、密码哈希、登录尝试与历史 | 满足基本身份鉴别与审计需求。 |

## §6 核心功能实现要点（约 3 分钟）

### 6.1 认证与用户体系

- 注册、登录、Token 刷新；角色 **teacher / admin / super_admin** 分流前端应用与接口权限。
- **超级管理员** 通过脚本初始化；验证码、登录失败锁定等由配置项与独立表支撑（详见 `docs/DATABASE.md`）。

### 6.2 视频处理与检测

- 视频入库后更新状态（上传中、处理中、完成、失败）。
- **抽帧**按配置间隔进行，调用 **多模型检测器**（`MultiModelDetector`）对单帧推理，得到类别与框；无模型时可走模拟数据保证链路可演示。
- **行为分类器**将原始检测映射为更贴近教研表述的师生行为标签；部分行为（如举手）带 **人数与去重** 等业务规则，减少帧级噪声对统计的影响。

### 6.3 时间聚合（答辩高频点）

- 采用 **固定长度、非重叠时间窗**（当前实现为 **10 秒**）：用视频时间戳对齐到窗起点，在每桶内对检测类别与分类行为做 **计数累加**。
- 结果写入 **`analysis_timeline`**；整课汇总与指标写入 **`analysis_summary`**，供图表与建议使用。详细说明见 **`docs/TIME_AGGREGATION.md`**。

### 6.4 行为分析与展示

- 提供整课汇总、时间线、异常、教学模式与相关性等 API；前端 **行为分析页** 用时间线构造趋势、热力图与模式推断等可视化。
- 分析服务中对时间序列可做 **滞后相关** 等统计（与 10 秒窗口径一致）。

### 6.5 教学建议（双通道）

- **综合诊断**：读取 **`AnalysisSummary`** 中行为占比与次数，用 **阈值规则** 生成结构化建议（如讨论占比偏低、低头率偏高、举手偏少等），延迟低、可解释。见 **`docs/教学建议与优化-答辩说明.md`**。
- **AI 教学顾问**：**LangGraph** 工作流从数据库拉取摘要与时间线等，拼入系统提示词，调用 **通义千问** 生成多轮回复；与规则建议 **互补**（稳定 vs 灵活）。

### 6.6 课表与辅助功能

- **课表** 与 **用户** 关联，支持批量导入或脚本导入；首页 **智能课表** 依赖后端课表 API 与缓存策略（参见 README 常见问题）。

### 6.7 数据库与建表

- 模型定义在 **`backend/app/models/`**；启动时 **`init_db()`** 执行 **`metadata.create_all`**。表包括用户、视频、三类分析表、课表、验证码与登录审计等。详见 **`docs/DATABASE.md`**。

## §7 创新点与工作总结（约 1 分钟）

**可强调的亮点**（按实际论文表述微调）：

1. **课堂场景闭环**：从上传到检测、聚合、可视化与建议，链路完整，数据同源。
2. **多模型 + 业务规则**：检测与后处理规则结合，使统计更贴近「教研语言」。
3. **可解释与可扩展**：结构化建议规则清晰；大模型侧明确「只基于当前课堂数据」的上下文边界。
4. **分级权限与双前端**：教学业务与超管运维分离，符合实际部署习惯。

**不足与展望**：例如精彩片段接口可完全数据驱动、跨课对比前后端路径需统一、规则阈值可配置化、数据库可换 PostgreSQL 以支撑更高并发等。

## §9 演示建议顺序（现场操作）

1. 启动 **后端** `python run.py`（8000），再启动 **Streamlit** 普通端（8501），必要时 **超管端**（8502）。
2. **登录**教师或管理员 → **首页**课表与入口。
3. **视频上传**（或选用已分析完成的视频）→ 展示 **进度与完成状态**。
4. **行为分析页**：时间趋势、模式或热力图（择 1～2 张图讲清数据来源）。
5. **教学建议页**：先看 **综合诊断**，再可选 **AI 顾问** 提问一句（需配置大模型 Key 与网络）。
6. 若有准备：**检测可视化视频**（`visualize_detection.py` 输出）可离线播放，展示模型框与类别。

## §12 附录：答辩提问备选答法（精简）

**Q：为什么用 Streamlit 而不是 Vue/React？**  
A：毕设周期内优先验证业务与算法闭环；Streamlit 与 Python 数据分析一致，图表集成快；未来可将 API 保留、仅替换前端。

**Q：检测准确率如何保证？**  
A：训练与验证在独立 yolo-vheat 工程完成；本系统侧重 **工程集成与统计口径**；答辩可展示验证集指标与可视化样例。

**Q：时间窗为什么取 10 秒？**  
A：在可解释性与平滑性之间折中；窗长可配置化，也可在展示层做二次平滑（见 TIME_AGGREGATION 文档）。

**Q：大模型会不会胡说？**  
A：结构化建议不依赖大模型；Agent 侧将 **分析结果文本**注入上下文，并限制为教学建议场景，降低脱离数据的概率，必要时可加重试与模板约束。

**Q：数据库为何用 SQLite？**  
A：单机演示与部署成本低；ORM 下迁移成本低，生产可换服务端数据库。

**Q：系统安全吗？**  
A：具备 JWT、哈希密码、登录限制与历史记录；生产需 HTTPS、密钥与 SMTP 等全部走环境变量，并替换默认密钥与演示账号密码。
