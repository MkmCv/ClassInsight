# 教学建议与课堂优化 — 答辩说明文档

## 1. 功能定位

「教学建议」面向**已完成行为分析**的课堂录像，在 **Streamlit 页面「💡 教学建议」** 中向教师提供两类能力：

| 能力 | 特点 | 适用场景 |
|------|------|----------|
| **综合诊断报告** | 基于分析结果的**规则化**建议，结构固定、可解释 | 快速浏览改进方向、答辩展示「数据驱动决策」 |
| **AI 教学顾问** | **大模型**结合课堂数据的多轮对话 | 追问细节、个性化表述、开放式建议 |

二者共用同一套上游数据：**视频处理任务**产出的 **`AnalysisSummary`、时间线、异常等**，不在此重复训练检测模型。

## 3. 综合诊断：规则化教学建议

### 3.1 接口与代码位置

- **路由**：`GET /api/v1/optimization/{video_id}/recommendations`
- **实现文件**：`backend/app/api/v1/endpoints/optimization.py` → `get_recommendations`

### 3.2 数据依据

- 读取 **`AnalysisSummary`**（与 `video_id` 对应，且视频状态为**已分析完成**）。
- 使用 `summary_json` 中的 **`behavior_summary`**：各类行为的 **`percentage`、`count`** 等（由 `video_processor` 在分析流水线末尾写入）。

### 3.3 规则逻辑（答辩可口述）

当前实现为**阈值 + 模板**，每条建议包含：`type`、`priority`、`title`、`description`、`suggested_actions`。

| 条件（摘自实现思路） | 建议方向 |
|----------------------|----------|
| `discuss` 占比 &lt; 15% | 增加互动、讨论与提问类行动项 |
| `BowHead` 占比 &gt; 30%（更高则提高优先级） | 关注注意力、节奏与多媒体等 |
| `hand-raising` 次数 &lt; 5 | 提升参与度、激励发言类行动项 |

若**未触发任何规则**，则返回一条**默认正向建议**（「保持良好教学状态」），避免页面无内容。

### 3.4 设计说明（答辩话术）

- **优点**：与具体模型输出解耦一层，规则清晰、延迟低、不依赖外部 LLM。
- **局限**：阈值与文案需随学科场景迭代；未覆盖的维度不会自动生成新类型建议。

## 4. 教学能力雷达与其它 Tab1 数据

### 4.1 雷达图

- **接口**：`GET /api/v1/optimization/{video_id}/radar`
- **实现**：同文件 `get_radar_chart_data`，在 `interaction_rate`、`attention_rate` 及 `behavior_summary` 中 **举手、guide、screen** 等字段上换算为多维得分，供前端 **Plotly 极坐标图** 展示。

### 4.2 精彩片段（当前实现状态）

- **接口**：`GET /api/v1/optimization/{video_id}/highlights`
- **现状**：后端以**示例数据**为主，代码注释标明**宜改为基于时间线/汇总的真实筛选**；答辩若被问到，可如实说明「接口已预留，片段质量模型待接」。

### 4.3 跨课次对比

- **接口**：`GET /api/v1/optimization/compare?video_ids=1,2,3`
- **作用**：拉取多个已完成视频的 `interaction_rate`、`attention_rate`、`engagement_score` 等，并计算简单**趋势**（升/降/稳）。
- **维护提示**：前端若请求路径与后端不一致，对比数据可能为空、界面回退默认演示数据；答辩前应**统一为 `/optimization/compare` 并传入合法 `video_ids`**。

## 5. AI 教学顾问：大模型 + 课堂上下文

### 5.1 接口与权限

- **路由**：`POST /api/v1/agent/chat`
- **实现文件**：`backend/app/api/v1/endpoints/agent.py`
- **校验**：当前用户对该 **`video_id`** 有权访问，且视频**分析已完成**；取对话中**最后一条用户消息**及历史，调用 Agent。

### 5.2 工作流（LangGraph）

- **图定义**：`backend/app/agent/graph.py`
- **典型流程**：
  1. **加载分析数据**（`load_analysis_data`）：通过 `tools.py` 异步读取 **`get_behavior_summary`、`get_behavior_timeline`、`get_anomaly_events`**，格式化为 **`analysis_context`** 文本。
  2. **生成回复**（`generate_response`）：系统提示词 + **当前课堂数据** + 多轮用户消息，调用 **阿里云 DashScope 通义千问**（如 `qwen-plus`）生成自然语言回答。

### 5.3 答辩表述建议

- **RAG 与否**：当前是「**把结构化分析结果注入 Prompt**」，而非向量检索文档；可称为 **Tool-augmented / 数据增强的对话**，便于老师理解。
- **与 Tab1 关系**：Tab1 **稳定、可审计**；Tab2 **灵活、可追问**，两者互补。

### 5.4 配置与安全

- API Key 宜通过环境变量 **`DASHSCOPE_API_KEY`** 配置；**切勿**在答辩材料或公开仓库中粘贴密钥。生产环境应轮换密钥并限制调用额度。

## 6. 前端页面要点

- **文件**：`System/frontend/pages/4_💡_教学建议.py`
- **行为**：侧边栏选择**已完成**视频；Tab1 请求 recommendations / highlights / 对比相关接口；Tab2 将 `video_id` 与 `messages` 发往 `/agent/chat`。
- **缓存**：列表与建议等使用 `st.cache_data` 短期缓存，减轻重复请求。

## 7. 答辩常见问题（简答）

**Q：教学建议从哪里来？**  
A：一类来自 **AnalysisSummary 上的规则引擎**；另一类来自 **通义千问**，输入是同一份分析摘要、时间线与异常信息。

**Q：如何保证建议可信？**  
A：结构化建议绑定**可复现的指标与阈值**；大模型侧强调**基于给定数据回答**，并在工程中通过权限与「仅已完成分析的视频」限制上下文范围。

**Q：和「行为分析」页的关系？**  
A：行为分析侧重**展示与统计**；教学建议侧重**解读与行动项**，数据同源、侧重不同。

**Q：还有哪些可改进点？**  
A： highlights 真实化、对比接口与前端路径统一、规则与阈值可配置化、按学科/学段的策略包等。
